# 个人实训总结报告

## 一、项目概述

本次实训我独立完成了一个全栈电商网站项目——"优选商城"。该项目是一个功能完整的单页应用（SPA），用户可以在网站上浏览商品、按分类筛选和搜索商品、查看商品详情并选择规格、加入购物车、提交订单、选择虚拟支付方式完成付款，以及查看历史订单和个人信息。

项目的前端采用纯 HTML + CSS + JavaScript 实现，没有使用任何前端框架，通过 hash 路由实现页面切换，所有数据通过 fetch API 从后端获取。后端采用 Python Flask 框架，配合 Flask-SQLAlchemy ORM 操作 SQLite 数据库，通过 RESTful API 向前端提供数据服务。整个项目前后端分离，前端作为静态文件由 Flask 服务器统一托管和提供。

目前项目共实现了 **7 个页面路由**（首页、商品详情、购物车、订单列表、个人主页、支付页、支付结果页）、**12 个后端 API 接口**（涵盖商品、购物车、订单三大模块）、**7 个商品分类**（数码电子、服饰穿搭、家居生活、美妆个护、运动户外、食品生鲜、图书文创）、**21 件商品**，每件商品均配有规格选择系统（如颜色、尺码、版本等），以及完整的虚拟支付流程。项目已成功部署到 Railway 云平台，可通过公网访问。

## 二、遇到的最大困难及解决过程

在整个开发过程中，我遇到了多个棘手的技术问题，以下是最关键的四个困难及其解决过程：

**困难一：CORS 跨域问题**

在前后端联调阶段，后端 API 用浏览器直接访问都能正常返回 JSON 数据，但前端页面一打开就报"加载商品失败"。我一开始以为是 API 地址写错了，检查了好几遍确认没问题。后来打开浏览器的开发者工具查看 Network 面板，发现请求返回了 CORS 错误。原因是 AI 在生成后端代码时，CORS 配置里只写了 `localhost:3000` 的白名单，而 Next.js 启动时 3000 端口被占用，自动切换到了 3001 端口，导致前端请求被后端拒绝。我花了大约一个小时排查这个问题，最终在后端 CORS 配置中将 origins 改为 `"*"` 才解决。这个问题让我意识到 AI 生成的代码不一定能适配实际的运行环境，需要开发者自己根据实际情况调试。

**困难二：Next.js 静态导出不支持 use client**

项目初期前端使用 Next.js 框架开发，但在准备部署到 Railway 时，我需要将 Next.js 项目导出为纯静态文件。然而执行 `next export` 时报错，因为项目中使用了 `'use client'` 指令（用于标记客户端组件），而 Next.js 的静态导出功能不支持这一特性。这意味着我之前基于 Next.js 的整个前端代码都无法直接部署为静态站点。经过分析，我做出了一个重要决定：放弃 Next.js，将整个前端改写为一个纯 HTML/CSS/JavaScript 的单页应用。所有页面逻辑集中在一个 `index.html` 文件中，用 hash 路由管理页面切换，用 fetch API 调用后端接口。这个改动虽然工作量大，但最终大大简化了部署流程，前端只需作为静态文件由 Flask 服务器提供即可。

**困难三：btoa() 不能处理中文字符**

这是整个开发过程中最隐蔽也最棘手的 bug。前端页面打开后商品列表完全无法加载，页面只显示"加载失败"。后端 API 返回 200 状态码，数据也正常，问题出在前端处理数据的过程中。经过逐步排查，我发现是 AI 生成的代码中使用了 `btoa()` 函数来处理图片的 data URI，而商品描述中包含中文字符。`btoa()` 是 JavaScript 的 Base64 编码函数，但它只能处理 Latin1 字符集（单字节字符），遇到中文等多字节字符会直接抛出异常 `Failed to execute 'btoa' on 'Window': The string to be encoded contains characters outside of the Latin1 range`。由于这段异常未被 try-catch 捕获，导致整个商品渲染流程中断，页面显示为空白或错误信息。更麻烦的是，AI 一开始给的修复方案是使用 `btoa(unescape(encodeURIComponent(text)))` 的技巧来绕过编码限制，但这种方法并没有解决根本问题——根本原因是不该用 btoa 来处理包含中文的字符串。最终我的解决方案是彻底移除 btoa 相关逻辑，图片直接使用后端返回的 `image_url` 路径显示，中文描述用 innerHTML 正常渲染。这个 bug 充分说明了 AI 生成的代码可能在某些边界条件下完全不可用，必须经过充分的测试。

**困难四：支付结果页路由解析 bug**

在实现虚拟支付功能后，支付成功跳转到支付结果页时页面显示 404。URL 格式为 `#/payresult?status=success&orderId=3&amount=299`。我一开始以为只是路由的 switch 语句里缺少 `payresult` 的 case 分支，加了之后发现仍然不行。后来通过 `console.log` 打印路由解析函数的返回值，发现 `page` 变量被解析成了 `"payresult?status=success&orderId=3&amount=299"`——整个 query string 被当作了路由路径的一部分。根本原因是 AI 编写的 `getRoute()` 函数在解析 hash 路由时，没有先将 `?` 后面的查询参数与路由路径分离。我手动在函数中添加了 `indexOf('?')` 的判断逻辑，在分割路由之前先把 query string 提取并保存，然后只对路径部分进行路由匹配。修复后支付结果页才能正常渲染。

## 三、AI 辅助编程的心得体会

通过本次实训项目，我对 AI 辅助编程有了深刻的认识。AI 在代码生成方面的能力确实令人印象深刻：当我向 AI 清晰描述需求后，它能在几分钟内生成一套完整的代码框架，包括数据库模型定义、API 路由实现、前端页面结构和样式等。对于搭建项目骨架、编写重复性高的模板代码这类工作，AI 的效率远超人工。

但 AI 辅助编程也有明显的局限性。最大的问题是 AI 生成的代码经常"看起来没问题，实际上跑不了"。前面提到的 CORS 跨域配置、btoa 中文编码、路由解析等 bug，都是 AI 生成的代码在特定场景下出现的问题。AI 没有在真实的运行环境中测试代码，也无法预判部署环境中可能出现的意外情况（如端口被占用、环境变量缺失等）。如果开发者完全信任 AI 生成的代码而不进行充分的测试和调试，项目很可能在关键时刻出问题。

我认为使用 AI 辅助编程的正确方式是：让 AI 负责"搭骨架"——生成项目结构、基础代码和重复性逻辑，然后由开发者自己来"填血肉"——调试错误、处理边界情况、优化用户体验、完善业务逻辑。在本项目中，大约 50% 的代码是由 AI 生成的（包括后端模型、API 路由、前端页面初始结构、CSS 样式等），另外 50% 则是我自己在调试和优化过程中编写和修改的（包括 bug 修复、交互逻辑调整、规格系统实现、支付流程完善、图片处理等）。

具体来说，开发者在 AI 辅助编程中的核心角色包括：第一，**需求描述者**——需要准确、详细地向 AI 描述功能需求和技术约束；第二，**测试工程师**——必须对 AI 生成的每一部分代码进行功能测试和边界测试；第三，**调试专家**——当 AI 的代码出现问题时，需要有能力定位根因并修复；第四，**架构决策者**——在技术方案出现分歧时（如 Next.js vs 纯 HTML），需要做出合理的技术选型判断。这些能力是 AI 无法替代的，也是本次实训中我最深刻的收获之一。

## 四、个人在项目中的具体贡献

在本项目中，我作为主要负责人完成了以下具体工作：

1. **设计了项目整体架构和技术选型**：确定了 Flask 后端 + 纯 HTML 前端 + SQLite 数据库的技术方案，设计了 RESTful API 接口规范和数据库表结构。
2. **编写了完整的后端 API 代码**：在 AI 辅助下完成了 models.py、app.py、config.py、seed_data.py 四个核心后端文件，涵盖 4 个数据模型和 12 个 API 接口。
3. **排查并修复了多个部署和联调问题**：包括 CORS 跨域配置错误、Railway 端口绑定问题、Dockerfile 构建失败等，成功将项目部署到 Railway 云平台。
4. **完成了前端技术方案的重大调整**：当发现 Next.js 静态导出不支持 use client 后，果断决定放弃 Next.js，将整个前端改写为纯 HTML/CSS/JavaScript 单页应用，涉及全部页面的重写。
5. **定位并修复了 btoa 中文编码 bug**：这个隐蔽的 bug 导致商品列表完全无法加载，通过逐步排查最终定位到 btoa 无法处理多字节字符的根因，并重写了图片处理逻辑。
6. **修复了价格显示 NaN 和人民币符号乱码问题**：添加了防御性的 parseFloat 处理，将 CSS 伪元素中的 ¥ 符号改为 JS 中的 Unicode 转义输出。
7. **设计并实现了商品规格选择系统**：改造了后端 Product 和 CartItem 模型支持 JSON 格式的规格存储，在前端实现了规格选择器 UI、选中状态管理和规格校验逻辑。
8. **实现了完整的虚拟支付流程**：包括支付方式选择 Modal、模拟支付过程（带 loading 状态）、支付成功/失败结果页，以及从下单到支付到结果展示的完整页面流转。
9. **为 21 件商品生成了 AI 产品图片**：逐一使用 AI 图片生成工具为每件商品生成匹配的产品图片，替换了原来依赖 Unsplash 外链的图片方案，大幅提升了国内访问速度。
10. **新增了 4 个商品分类和 10 件商品**：扩展了项目的商品覆盖范围，从 3 个分类 11 件商品扩展到 7 个分类 21 件商品，更新了种子数据、前端分类映射和商品图片。
11. **编写和维护了项目文档**：包括 API 接口文档、代码审查报告、AI 辅助编程记录（prompt_log.md）和本篇个人总结报告，确保项目有完整的技术文档支撑。
12. **完成了项目的线上部署和持续维护**：配置了 Dockerfile 和 Railway 部署环境，处理了端口绑定、静态文件服务、数据库持久化等部署细节，确保线上服务稳定运行。

## 五、总结

通过本次 AI 辅助编程与工程化实训，我在多个方面获得了显著的成长和收获。

在技术层面，我对全栈 Web 开发有了更加系统和深入的理解。从后端的数据库模型设计、ORM 操作、RESTful API 开发，到前端的页面渲染、路由管理、异步数据请求，再到前后端联调、跨域处理、云平台部署，每一个环节我都亲自参与并解决了实际问题。这些实践经验远比课堂上的理论学习更加生动和深刻。

在调试能力方面，本次项目的多次 bug 排查经历极大地锻炼了我的问题定位和分析能力。从 CORS 跨域到 btoa 编码，从路由解析到价格显示，每一次问题的排查都让我学会了更系统地使用浏览器开发者工具（Network 面板、Console 面板）、更善于通过日志和断点定位问题根因、更注重代码的防御性编程和边界情况处理。

在 AI 协作方面，我学会了如何在 AI 辅助下高效完成开发任务。关键的认识是：AI 是工具而非替代者，它能帮你快速生成代码初稿，但代码的正确性、健壮性和用户体验最终取决于开发者自己的判断和努力。用 AI 之前要先把需求想清楚，用 AI 之后要把代码测透彻，这两者缺一不可。

总的来说，本次实训不仅让我完成了一个功能完整的电商项目，更重要的是培养了我独立解决技术问题、合理利用工具提升效率的工程化思维和实践能力。
